## OAuth 2.0 Core Concepts

### Access Token
**Definition**: Short-lived credential (typically 1-hour) used to access protected resources on behalf of the user.
**Context**: Obtained after successful OAuth flow, included in Authorization header of API requests.
**Example**: `Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...`
**Added In**: Planning phase
**Related Terms**: Refresh Token, Bearer Token

### Authorization Code
**Definition**: One-time code issued by OAuth provider after user consent, exchanged for access token.
**Context**: Prevents access token from being exposed in browser redirect URL.
**Lifetime**: Single-use, expires in 10 minutes (Google default)
**Added In**: Planning phase
**Related Terms**: Authorization Code Flow, PKCE

### Authorization Code Flow
**Definition**: Most secure OAuth 2.0 flow for web applications with server-side component.
**Steps**:
1. Redirect user to OAuth provider's authorization URL
2. User authenticates and grants consent
3. Provider redirects back with authorization code
4. Server exchanges code for access token (server-to-server)
5. Access token used for API requests
**Added In**: Planning phase
**Related Terms**: OAuth 2.0 Flow, PKCE

### Refresh Token
**Definition**: Long-lived credential used to obtain new access tokens without user interaction.
**Context**: Stored securely on server, rotated on each use (refresh token rotation).
**Lifetime**: 90 days (configurable), reset on each successful refresh
**Added In**: Planning phase
**Related Terms**: Access Token, Refresh Token Rotation

## Security Patterns (Added in Research Phase)

### PKCE (Proof Key for Code Exchange)
**Definition**: Security extension to OAuth 2.0 protecting against authorization code interception attacks.
**Pronunciation**: "pixie"
**How It Works**:
1. Client generates random `code_verifier` (43-128 characters)
2. Client creates `code_challenge` = BASE64URL(SHA256(code_verifier))
3. Authorization request includes `code_challenge`
4. Token exchange includes `code_verifier`
5. Provider validates: SHA256(code_verifier) == code_challenge
**Added In**: Research phase
**Related Terms**: Code Verifier, Code Challenge
**References**: RFC 7636

### Code Verifier
**Definition**: Cryptographically random string (43-128 chars) generated by client for PKCE.
**Format**: `[A-Z] / [a-z] / [0-9] / "-" / "." / "_" / "~"`
**Example**: `dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk`
**Storage**: Never sent to browser, kept server-side only
**Added In**: Research phase
**Related Terms**: PKCE, Code Challenge

### Code Challenge
**Definition**: Base64-URL-encoded SHA-256 hash of code verifier, sent in authorization request.
**Calculation**: `BASE64URL(SHA256(ASCII(code_verifier)))`
**Example**: `E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM`
**Purpose**: Provider stores this, validates against code_verifier later
**Added In**: Research phase
**Related Terms**: PKCE, Code Verifier

### httpOnly Cookie
**Definition**: Cookie flag preventing JavaScript access, mitigates XSS attacks.
**Usage**: Session tokens stored in httpOnly cookies
**Browser Support**: All modern browsers
**Trade-off**: Cannot read cookie client-side (good for security, bad for some SPAs)
**Added In**: Research phase
**Related Terms**: Secure Cookie, SameSite Cookie

### Secure Cookie
**Definition**: Cookie flag ensuring transmission only over HTTPS, never HTTP.
**Usage**: All session and auth-related cookies
**Enforcement**: Browser refuses to send cookie over HTTP
**Development**: Set to false in local dev (http://localhost)
**Added In**: Research phase
**Related Terms**: httpOnly Cookie, SameSite Cookie

### SameSite Cookie
**Definition**: Cookie flag controlling cross-site request behavior, CSRF mitigation.
**Values**:
- `Strict`: Never sent with cross-site requests
- `Lax`: Sent with top-level navigation (clicked links), not subrequests
- `None`: Always sent (requires Secure flag)
**Our Choice**: `Lax` (balance security and usability)
**Added In**: Research phase
**Related Terms**: CSRF, httpOnly Cookie

## Implementation Terms (Added in Design Phase)

### Refresh Token Rotation
**Definition**: Security pattern where refresh token is invalidated and replaced on each use.
**Benefits**:
- Limits impact of token theft (stolen token only works once)
- Enables detection of token reuse (alerts on old token usage)
- Recommended by OAuth 2.0 Security Best Practices
**Implementation**: Store `previous_token_hash` for grace period
**Added In**: Planning phase, refined in Design phase
**Related Terms**: Refresh Token, Token Revocation

### Session Fingerprinting
**Definition**: Binding session to specific client characteristics (User-Agent, IP) to detect session hijacking.
**Checks**:
- User-Agent must remain constant
- IP address changes trigger re-authentication (configurable)
**Trade-offs**: Can break legitimate use (VPN switches, mobile networks)
**Our Implementation**: User-Agent check only (less strict)
**Added In**: Design phase
**Related Terms**: Session Security, Session Hijacking

### Token Revocation
**Definition**: Invalidating access/refresh tokens before natural expiration.
**Triggers**:
- User logs out
- Password change
- Admin action (ban user)
- Security incident
**Implementation**: Delete from `refresh_tokens` table, add to revocation list
**Added In**: Design phase
**Related Terms**: Refresh Token, Logout

### OAuth Scopes
**Definition**: Permissions requested from OAuth provider, controls access level.
**Google Scopes Used**:
- `openid`: Required for OpenID Connect
- `profile`: User's basic profile info (name, picture)
- `email`: User's email address and verification status
**Format**: Space-separated string
**Example**: `"openid profile email"`
**Added In**: Planning phase, detailed in Design phase
**Related Terms**: OAuth 2.0, OpenID Connect

## Infrastructure Terms (Added in Execution Phase)

### Redis Session Store
**Definition**: Redis-backed session storage for Express applications.
**Library**: `connect-redis` (official Redis session store for Express)
**Benefits**:
- Persistence across server restarts
- Horizontal scalability (shared session state)
- Fast lookups (sub-millisecond)
**Schema**: Key-value store, key = `sess:{session_id}`, value = JSON session data
**Added In**: Planning phase, implemented in Execution phase
**Related Terms**: Session Storage, Redis

### Passport.js
**Definition**: Authentication middleware for Node.js supporting 500+ strategies.
**Architecture**: Pluggable strategies (passport-google-oauth20, passport-local, etc.)
**Integration**: Express middleware, `req.user` populated on success
**Session**: Serializes/deserializes user to/from session
**Added In**: Research phase, implemented in Execution phase
**Related Terms**: Authentication Middleware, OAuth Strategy

### passport-google-oauth20
**Definition**: Passport.js strategy for Google OAuth 2.0 authentication.
**Configuration**:
- Client ID (from Google Cloud Console)
- Client Secret (from Google Cloud Console)
- Callback URL (where Google redirects after auth)
**Supports**: OAuth 2.0 with PKCE, OpenID Connect
**Added In**: Research phase
**Related Terms**: Passport.js, Google OAuth

### Helmet.js
**Definition**: Express middleware for setting security-related HTTP headers.
**Headers Configured**:
- Content-Security-Policy (CSP)
- Strict-Transport-Security (HSTS)
- X-Frame-Options (clickjacking protection)
- X-Content-Type-Options (MIME sniffing protection)
**Usage**: `app.use(helmet())`
**Added In**: Execution phase
**Related Terms**: Security Headers, Express Middleware

### Winston
**Definition**: Logging library for Node.js with multiple transports (file, console, remote).
**Usage**: Structured JSON logging for auth events
**Log Levels**: error, warn, info, debug
**Transports Used**: Console (dev), File (production)
**Added In**: Execution phase
**Related Terms**: Structured Logging, Observability

## Database Terms (Added in Design Phase)

### users Table
**Definition**: Primary table storing user account information.
**Schema**:
```sql
CREATE TABLE users (
  id UUID PRIMARY KEY,
  email TEXT UNIQUE NOT NULL,
  email_verified BOOLEAN DEFAULT false,
  google_id TEXT UNIQUE,
  user_metadata JSONB,
  created_at TIMESTAMP DEFAULT NOW(),
  last_login_at TIMESTAMP
);
```
**Indexes**: email, google_id
**Added In**: Design phase

### refresh_tokens Table
**Definition**: Stores refresh tokens with audit trail for rotation.
**Schema**:
```sql
CREATE TABLE refresh_tokens (
  id UUID PRIMARY KEY,
  user_id UUID REFERENCES users(id) ON DELETE CASCADE,
  token_hash TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  last_used_at TIMESTAMP,
  expires_at TIMESTAMP NOT NULL,
  revoked_at TIMESTAMP
);
```
**Indexes**: user_id, token_hash, expires_at
**Cleanup**: Cron job deletes revoked/expired tokens daily
**Added In**: Design phase

### sessions Table
**Definition**: Redis-backed session storage (not PostgreSQL table).
**Key Format**: `sess:{session_id}`
**Value**: JSON object with user_id, csrf_token, created_at, expires_at
**TTL**: 7 days (Redis auto-delete on expiration)
**Added In**: Design phase, implemented in Execution phase

## Security & Compliance (Added in Review Phase)

### Defense in Depth
**Definition**: Layered security approach where multiple defenses protect against threats.
**Our Layers**:
1. HTTPS (transport encryption)
2. PKCE (authorization code protection)
3. httpOnly cookies (XSS mitigation)
4. Token encryption at rest (Redis compromise mitigation)
5. Session fingerprinting (hijacking detection)
6. Rate limiting (brute force prevention) - to be added
**Added In**: Research phase, validated in Review phase
**Related Terms**: Security Best Practices

### OWASP
**Definition**: Open Web Application Security Project, publishes security standards.
**Relevant Guides**:
- Authentication Cheat Sheet
- Session Management Cheat Sheet
- OAuth 2.0 Cheat Sheet
**Compliance**: Implementation follows OWASP recommendations
**Added In**: Research phase
**Related Terms**: Security Standards, Best Practices

### Rate Limiting
**Definition**: Restricting number of requests from client within time window.
**Purpose**: Prevent brute force attacks on auth endpoints
**Recommendation**: 5 attempts per 15 minutes per IP
**Library**: `express-rate-limit` (to be added)
**Status**: Medium-priority enhancement (not blocking production)
**Added In**: Review phase
**Related Terms**: Brute Force Protection, Security

## Acronyms & Abbreviations

- **CSP**: Content Security Policy

- **SSO**: Single Sign-On
