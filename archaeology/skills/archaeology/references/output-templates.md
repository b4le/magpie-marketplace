# Output Templates

Shared output templates for archaeology domain findings. All templates use YAML frontmatter and consistent markdown structure.

---

## Frontmatter Template {#frontmatter}

All output files MUST include frontmatter with these fields:

```yaml
---
generated: YYYY-MM-DD
project: ProjectName
domain: domain-id
source_count: N
confidence: high|medium|low
last_search_date: YYYY-MM-DD
---
```

**Field descriptions:**

- **generated** - Date this document was created (ISO format)
- **project** - Human-readable project name
- **domain** - Domain ID from `registry.yaml`
- **source_count** - Number of source files/conversations analyzed
- **confidence** - Assessment of completeness (optional)
  - `high` - Exhaustive search, all major patterns captured
  - `medium` - Representative sample, may have gaps
  - `low` - Initial exploration, incomplete coverage
- **last_search_date** - When the search was last run (for staleness tracking)

**Example:**
```yaml
---
generated: 2026-02-26
project: E-Commerce Platform
domain: rest-apis
source_count: 47
confidence: high
last_search_date: 2026-02-26
---
```

## Usage Notes

### Choosing Templates

**README:** Always create for every domain search
**Patterns:** Use when architectural decisions are found
**Examples:** Use when concrete code implementations are found
**Prompts:** Use when domain involves conversational patterns
**Topology:** Use when system architecture is complex
**No Results:** Use when search returns empty

### Customizing Templates

**Domain-specific sections:**
Add sections specific to your domain at the end, before metadata.

**Anchors:**
All templates include `{#anchor}` tags for cross-referencing. Maintain these.

**Metadata:**
Always include the metadata section at the bottom. This enables:
- Staleness detection
- Aggregation across domains
- Search result filtering

### Linking Between Files

Use relative links:
```markdown
See [pattern name](./patterns.md#pattern-name)
See [example](./examples.md#example-name)
```

### Version Control

**Frontmatter versioning:**
```yaml
---
generated: 2026-02-26
project: MyProject
domain: rest-apis
version: 2  # Increment on major changes
changelog:
  - "2026-02-26: Added 5 new patterns"
  - "2026-01-15: Initial extraction"
---
```

**File naming for iterations:**
- `patterns.md` - Current version
- `patterns-v1.md` - Previous version (archive)

## Validation

### Required Elements

All output files MUST have:
- [ ] YAML frontmatter with all required fields
- [ ] H1 title matching template
- [ ] Metadata section at bottom
- [ ] Generated by attribution line

### Quality Checks

**Good output:**
- Concrete examples with file paths
- Dates and context for historical findings
- Clear categorization
- Cross-links between related content

**Poor output:**
- Vague descriptions without specifics
- No source attribution
- Generic examples that could apply to any project
- Broken links

### Automation

Check output quality with:
```bash
# Verify frontmatter
grep -L "^---" output/**/*.md

# Verify metadata section
grep -L "## Metadata" output/**/*.md

# Verify attribution
grep -L "Generated by archaeology skill" output/**/*.md
```

## Completion Display Templates

Terminal output shown to the user after each mode completes. These are **contract formats** — reproduce them character-for-character, substituting only `{variable}` placeholders.

**Format enforcement (applies to ALL completion displays):**
- MUST use the exact template below — no reformatting
- MUST NOT use ASCII box-drawing characters (`─│┌┐└┘├┤┬┴┼`)
- MUST NOT use emoji
- MUST NOT use markdown tables, markdown bold, or markdown headers
- MUST NOT use horizontal rules (`---`)
- MUST NOT reformat stat lines as bullets or numbered lists
- MUST NOT add decorative formatting beyond what the template specifies
- Each domain/item line MUST be indented with exactly 2 spaces

**Branding (applies to ALL completion displays):**

Resolve these from `references/branding.md` Variables section:
- **Title sigil:** Prefix the first line with `SIGIL_{MODE}` (e.g., `SIGIL_SURVEY` = `◈`)
- **Sign-off:** Append `SIGNOFF_TEMPLATE` as the final line, substituting `{mode}`

If the branding variables in `references/branding.md` change, all completion displays update automatically — do not hardcode sigils or sign-off text anywhere else.

```
  {d.id}  {d.signal}  score: {d.score}  ({d.sessions} sessions)

{SIGNOFF}
```

**Variables:**
- `d.*` — from `domain_scores` (S3), filtered to `d.signal !== 'none'`, sorted score-descending. One line per domain.
- `d.id` — domain ID from registry (NOT display name). `d.score` — use `.toFixed(1)`.
- `dive.*` — from `suggested_dives` (S5). One line per dive.
- If no domains have signal: replace the "Domains with signal:" block with `No domains with signal detected.`
- If no suggested dives: replace the "Suggested deep dives:" block with `No uncovered patterns detected.`

**If `--no-export`:** Use title `Archaeology Survey Complete (export skipped)`, omit Central paths.
